ADM - Netezza - Analytics

  1. Visão geral da Adm.
    2.

    As tarefas de administração do Netezza se dividem em 2 categorias, sendo elas:

    • System Administration - Gerenciamento do hardware, configurações de sistema, status do sistema, acessos, discos e outras tarefas.
    • Database Administration - Gerenciamento dos usuários das bases e seu conteúdo, como carregamento e manipulação de dados, consultas, restores de dados e controle de acessos/permissões aos dados e a quem os acessa.

    Em alguns ambientes, uma pessoa pode possuir ambas formas de administração, tanto de sistema como de databases, para o gerenciamento de todo o ambiente e suas tarefas. Em outros ambientes, várias pessoas podem compartilhar dessas responsabilidades, ou a separação específica em equipes para cada forma de administração. Além dessas permissões gerais, um usuário também pode possuir as permissões de database, a qual define como um usuário poderá se comportar diante de um banco de dados e suas tabelas, como permissões de criação de tabelas/bases, inserção/deleção de registros e etc.

    De forma geral, as tarefas de administração se ocupam nas seguintes categorias:

    • Instalação dos clients do Netezza
    • Gerenciamento do Appliance do Netezza
    • Gerenciamento de notigficações/eventos de sistema
    • Gerenciamento dos usuários e grupos
    • Gerenciamento das databases
    • Carregamento de dados
    • Backup e restores de bases/dados
    • Histórico de querys (logs)
  2. Usando a interface de administração do Netezza
    3.

    Esta etapa irá proporcionar um alto nível de descriçãao para a administração do Netezza através de interface e linhas de comando também ,assim como NzAdmin, Interface de administração web e comandos SQL para o Netezza. Além de descrever como realizar o acesso e usar dessa interface. Há a possibilidade de realizar as tarefas de gerenciamento do Netezza via linha de comando, para a a gerência do software/hardware e das bases usadas. Esses comandos são chamados de *nz commands

Muitos desses comandos são geridos junto a instação do kit client Netezza e podem rodar de forma remota no ambiente.

Enquanto alguns comandos podem operar e exibir informações sem os acessos requeridos, outro comandos e operações requerem que seja especificado o usuário e senha da base em questão.

  3. Startando o NzAdmin
    4.

    Este funciona como um gerenciador de administração de um servidor do Netezza, possuindo uma interface gráfica dinâmica que exibe informações de todo um servidor, desde suas configurações e disponibilidade até suas databases/tabelas e usuários que se encontram. Para que se tenha acesso, é necessário o host da máquina que está inserido e dos dados de usuário/senha. Após o log in ser efetuado, a interface será aberta com a página dos dados gerais do servidor. Perceba que há duas possíveis janelas sendo: system (Informações do servidor em sí) e Database (Informações das bases e tabelas presentes no servidor em questão).

    Como dito anteriormente, a primeira janela a ser exibida é a de system das informações do servidor, em seu lado direito é possível ver os dados gerais do servidor e no lado esquerdo a arvore de informações. Neste lado direito, é exibido uma visão geral do hardware, com imagem do sistema Netezza que podem conter 1 ou mais racks do applience. Estas imagens são interativas, logo ao posicionar o mouse sobre o servidor, são exibidas informações como ID do hardware e outros detalhes. Ao clicar na imagem do servidor, o usuário é redirecionado para uma página com mais informações detalhadas sobre aquele em questão (Além disso, no canto inferior direito, é possível ver o usuário que abriu esta sessão e o tempo de duração desde a abertura).

    Perceba que o status geral dos componentes são baseados em cores, as quais possuem uma descrição no próprio exibidor da interface. Essas cores indicam o status do um componente X (na primeira janela por exemplo, são exibidos os status do servidor). Na parte superior da interface, se encontra o menu de execuções do ambiente, possibilitando a execução de comandos via interface gráfica, como criação de base/tabela e etc, onde:

    • File > New - Allows you to create a database, table, view, materialized view, sequence, synonym, user or group. Available only in Database tab.

    • File > System - State Allows you to change the system state.

    • File > Reconnect - Reconnects to the Netezza system with a different hostname, address or user name.

    • File > Exit - Exits the application.

    • View > Toolbar - Shows/hides the application toolbar.

    • View > Status - Bar Shows/hides the status bar.

    • View > System - Objects Shows/hides system tables and views and applies to object privilege lists in the Object Privileges window.

    • View > SQL - Statements Displays the SQL Window that shows a subset of the SQL commands NzAdmin has used in this session.

    • View > Refresh - F5 Refreshes the entire view — What is refreshed depends on whether you are viewing the System or Database section of the NzAdmin tool.

    • Tools > Workload Management - Performance — Displays Summary, History, and Graph Workload Management information.

    • - Settings — Displays the System Defaults that you can use to set the limits on session timeout, row set, query timeout, and session priority and the Resource Allocation that you can use to specify resource usage among groups.

    • Tools > Table Skew - Table Skew — Displays any tables that meet or exceed a specified skew threshold.

    • Tools > Table Storage - Table Storage — Displays table and materialized view storage usage by database or by user.

    • Tools > Query History Configuration - Query History Configuration — Displays a window that you can use to create and alter query history configurations, as well as to set the current configuration.

    • Tools > Default Settings - Default Settings — Displays the materialized view refresh threshold.

    • Tools > Options - Preferences — Displays the Preferences dialog box that you can use to set the object naming preference and whether you want auto refresh.

    • Help > NzAdmin Help - Displays the online help for the NzAdmin tool.

    • Help > About NzAdmin - Displays the NzAdmin and Netezza revision numbers and copyright text.

    System Window

    Aqui serão exibidas as informações do hardware em sí, mostrando o andamento do servidor em questão e seus discos, assim como o status das SPU's e Data Slices.

    1. Página inicial - Informações gerais do servidor
      2.

      Nesta primeira janela em que o usuário se encontra, são exibidas os dados gerais do servidor em questão, sendo possível ver uma imagem do servidor (ao passar o mouse, é possível exibir a quantidade de racks e a ID do servidor). Além disso, é exibido também, o estado em que o banco de dados do servidor se encontra (database status) através de uma cor (a qual possui legenda ao lado), indicando se está OK, em estado crítico ou necessitando de atenção.

      Outras informações que são mostradas nesta janela, é o estado do sistema (system state), que define se o servidor se encontra fora ou não do ar. Assim como a Autenticação do serviço (Authentication), a qual mostra se é um serviço LOCAL ou REMOTE. Logo abaixo, um gráfico do armazenamento do disco é mostrado ao usuário (system disk status), sendo um gráfico pizza, com dados do armazenamento total no disco, quantidade de armazenamento usado e disponível também.

    2. Segunda Janela - Informações específicas do servidor
      3.

      Para alcançar esta segunda página, basta que o usuário clique na imagem do servidor exibida na primeira janela (As imagens do NzAdm são interativas), redirecionando o suário para a página SPA ID. Nela, são exibidas informações mais claras do servidor, começando por uma imagem mais clara do mesmo (front view) que exibe o status de cada disco presente no servidor (seguindo o mesmo conceito de cores citado anteriormente). Cada disco se encontra em uma SPU do servidor, essa SPU são compostas por FPGA's, CPU e memória associadas a quantidade N de discos de armazenamento no servidor. Portanto, assim como o status de cada disco, também é possível ver o status da(as) SPU's usadas pelo servidor, logo abaixo.

      Nas informações do disco (sendo exibidas ao posicionar o mouse sobre o mesmo) se referem a: ID, Serial Number e Enclosure ID(Gabiente) de cada disco. Além disso, ao clicar sobre um disco X, no canto mais a direita são exibidas suas informações de forma mais detalhada, como o tipo do disco, sua IP, modelo e etc (por default são exibidas neste lado, as informações de todos os discos sem especificação, sendo alterado apenas quando um disco específico é selecionado).

      O mesmo acontece com as SPU's do servidor, onde suas informações gerais são de: ID, SPA ID(servidor), Serial Number e IP Address. Assim como ao clicar sobre um SPU, suas informações detalhadas são exibidas no canto direito.

      Por fim, mais abaixo, é possível ver informações mais básicas, como o status do servidor (SPA status), Hardware ID, quantidade de racks e discos spare (discos em standby que serão usados como metidas de prevenção de risco, em caso de um disco X falhar, um spare disk é usado no lugar).

    3. Terceira Janela - Informações das SPU's
      4.

      Nesta terceira janela (janela dentro da SPA ID), são dadas as informações detalhadas de cada SPU do servidor (SPU Slot). Esses dados na parte superior (SPU Status/SPU Detail) são exatamente sobre a SPU em questão, mostrando a sua ID, status, SPA que se ecnontra, versão e etc. Logo abaixo, são exibidas as informações dos Data Slices, que mostram o status, partição, tamanho, discos suportados e etc.

      Esses Data Slices são administrados pelas SPU's em questão de performance.

    4. Quarta Janela - Informações gerais das SPU's
      5.

      Nesta janela SPU Units são exibidas as informações gerais das SPU's presentes no servidor, como seu status, rack em que se encontra, IP e etc.

    5. Quinta Janela - Data Slices
      6.

      Aqui são mostradas as informações específicas de todos os Data Slices presentes no servidor, as quais são exibidas extamanete da mesma forma na janela de informações das SPU's, que como dito anteriormente, são as SPU's que administram o funcionamento desses Data Slices em cada base.

      Os Data Slices funcionam como um "quebrador" de dados, onde de uma explicação geral, um dado é todo quebrado/fragmentado em várias partes visando a performance do Netezza para sua exibição, esse dado será jogado suas partes em cada disco e trabalhado. Para isso os Data Slices, que funcionam como o triturador dessesdados, aumentando a performance do Netezza.

    Database Window

    Nesta janela, serão exibidas as informações de cada banco de dados existente no servidor, assim como as informações das tabelas, usuários e etc. Através desta janela, é possível monitorar todo o servidor em questão (dependendo das permissões concedidas ao usuário), além de que também é possível ver o andamento de cada base e até mesmo, cada sessão aberta no servidor, como seu usuário, a base que está acessando, horário de acesso e até mesmo o comando gerado pela sessão em uma base X.

    De certa forma, é nesta janela que está a principal monitoração das bases do Netezza, onde um DBA por exemplo, poderá definir todas as regras e visualizar o andamento de todo o servidor Netezza. Além disso, é possível ver informações do histórico de backup de todo o servidor e também o histórico de restore's realizados no mesmo.

  4. Gerenciando um servidor Netezza
    5.

    Aqui serão descritas as maneiras de gerenciar os servidores e processos do Netezza, além do impacto gerado durante ações de started e stopped nas tarefas do Appliance. Outras informações que serão listadas neste capitulo, são sobre o comportamento dos chamados arquivos de log, onde encontra-los e mensagens de erro de troubleshooting. Diante disso, começaremos com a explicação e orientação dos estados do sistema (system state).

    • System State

      • O estado sistema Netezza se refere ao atual estado em que o appliance se encontra, onde normalmente se encontra online e operando normalmente. Porém, ebventualmente, esse serviço necessita ser parado (stopped) para algumas medidas/ações (como atualização, manutenção, necessidade, prevenção e etc). É possível gerenciar esses estado do sistema, através do comando nzstate, em que será exibida as informações gerais sobre o stado atual do sistema. Diante disso, os estados possíveis do sistema podem se encontrar em:

      • Online - Estado em que o Netezza se encontra totoalmente operacional (sendo o mais comum dos estados), onde o sustema se encontra pronto para processar queries. O sistema entra nete estado quando o Netezza é iniciado de algum modo (seja ele restardado ou não).

      • Offline - Estado em que o Netezza é interrompido, ainda conseguindo continuar a execução de queries já iniciadas e gerando erro para novas consultas. Sendo realizado após um comando de nzsystem offline e voltando à online através de comandos nzsystem resume/stop.

      • Paused - Estado em que o sistema se encontra em uma breve interrupção do servidor, onde o sistema continua a executar tarefas que já foram iniciadas, porém recusa qualquer outra que seja solicitada. Para isso, é ncessário o comando de nzsystem pause e retornando ao normal através do comando nzsystem resume/stop.

      • Down - Estado em que o sistema se encontra caso haja uma insuficiência por parte do hardware para as funções do mesmo (conseguindo informações através do comando nzstate -reason). Para o retorno de seu funcionamento, são necessárias reparações nos hardwares e em seguida o comando nzsystem resume.

      • Stopped - Estado em que o o sistema se encontra para medidas de tarefas necessárias (onde para suas execuções, o Netezza não pode estar no ar), como a instalação de um novo software. Sendo exigido através do comando nzsystem stop e retornando ao ar através do comando nzstart

        • Perceba que durante os comandos do usuário de nzsystem stop/pause/offline/restart são executados, o sistema permite a continuação e espera finalizar queries que já se encontram rodando terminar, para então realizar a parada geral do sistema (a não ser que seja dado junto ao comando, a instrução de -now, que irá parar o sistema imediatamente e irá ignorar as tarefas, mesmo que ainda não terminadas).

      Quando o software do Netezza está rodando (esta no ar), o sistema e as SPU's podem fazer a transição pelos seguintes estados transacionais: de curta duração (estados terminados em "ing", como Pausing, Discovering) e os estados de longa duração, onde o sistema transita sobre estes estados em casos de ações forçadas ou de mudança. Todos esses estado serão descritos a seguir:

      Estado Descrição
      Down The system has not been configured (there is no configuration information for the data slices to SPU topology) or there is not enough working hardware to operate the system even in failover. The SPUs can never be in this state.
      Discovered The SPUs and other components are discovered, but the system is waiting for all components to complete start-up before transitioning to the initializing state.
      Discovering The system manager is in the process of discovering all the system components that it manages.
      Going Offline The system is in an interim state going to offline.
      Going Offline (Now) The system is in an interim state going to offline now
      Going Pre-Online The system is in an interim state, going to pre-online.
      Initialized The system uses this state during the initial startup sequence.
      Initializing The system is initializing. You cannot execute queries or transactions in this state
      Missing The system manager has detected a new, unknown SPU in a slot that was previously occupied but not deleted.
      Offline (Now) This state is similar to offline, except that the system stops user jobs immediately during the transition to offline.
      Online The system is running normally. It can service requests
      Paused The system is paused. You cannot run user jobs.
      Paused (Now) This state is similar to paused, except that the system stops user jobs immediately during the transition to paused.
      Pausing The system is transitioning from online to paused. During this state no new queries or transactions are queued, although the system allows current transactions to complete, unless you have specified the nzsystem pause -now command.
      nzrev Displays the current software revision for any Netezza software release.
      Pausing Now The system is attempting to pause due to a hardware failure, or the administrator entered the nzsystem pause -now command.
      Pre-Online The system has completed initialization. The system goes to the resume state.
      Resuming The system is waiting for all its components (SPUs, SFIs, and host processes) to reach the online state before changing the system state to online
      Stopped The system is not running. Note that commands assume this state when they attempt to connect to a system and get no response. The SPUs can never be in this state
      Stopped (Now) This state is similar to stopped, except that the system stops user jobs immediately during the transition to stopped.
      Stopping The system is transitioning from online to stopped.
      Stopping Now The system is attempting to stop, or the administrator entered the nzsystem stop -now command.
      Unreachable The system manager cannot communicate with the SPU because it has failed or it has been physically removed from the system

    • Gerenciamento o System State
      •

      É possível dar os comandos de nzstart/nzstop respectivamente parar iniciar/parar as operações do Netezza. Através do nzsystem são fornecidas opções adicionais de alteração de estado, como restartar o sistema e etc. Ao dar um comando de stop/start nas operações do Netezza, os storage groups continuam a executar, assim como tarefas de verificação de mídia e integridade. Porém, entenda que para que os comandos de mudança de estados sejam realizados e executados pelo sistema, é necessária a permissão de usuário a quem esteja dando o comando.

      • Estartando o sistema

        • Ao iniciar um sistema Netezza, é acionado ao sistema e aos processos de databases que estejam online, fazendo com que todo o sistema esteja pronto para rodar queries e outras taregfas. Este estado pode ser iniciado a partir do comando nzstart , invocando um startup ao servidor, onde mensagens de verificação ou erro podem ser geradas em casos de congigurações mal feitas ou algo do tipo.

        Tal comando só pode ser executado quando o host for logado ao usuário, não sendo possível executa-lo remotamente de sistemas client do Netezza.

      • Parando o sistema

        • Ao parar um sistema Netezza, tanto os processos de database como serviços de backup, queries e etc, são parados e não serão iniciados enquanto o estado permanecer. Normalmente, só é dado um comando de stop no servidor quando realmente necessário, como casos de administração específica, ou para a paragem crítica de uma tarefa. Parando o sistema, todo os processos do Netezza serão parados, aguardando aqueles que já foram iniciados, terminarem suas execuções.

      • Pausando o sistema

        • Certas tarefas, como backup de host, exigem que o sistema esteja pausado para sua execução. Ao pausar, o sistema enfileira as tarefas que seriam executadas em ordem de chamada ou prioridade, para posteriormente (ao ser retomado) executa-las devidamente. Por default, o sistema finaliza as queries e transações que já foram ativas até a sua retomação. Ao ser dado o comando nzsystem pause, uma pergunta de certeza é exibida ao usuário, necessitando responde-la para a sequência do procedimento.

      • Retornando o sistema

        • Este é o comando dado após a pausa do Netezza, fazendo com que o sistema saia de estado atual para retornar ao ar. o realizar isso, automaticamente todos os processos e transações serão retomados a suas execuções e continuaram (em alguns casos, há a chance de o sistema restartar uma tarefa, iniciando-a novamente).

      • Tornando o sistema offline

        • Tornar o sistema fora do ar, fará com que nenhuma transação ou tarefa seja iniciada eventualemnte, permitindo apenas que tarefas de gerenciamento continuem rodando, concluindo todas as transações e queries que já foram estartadas. Como na pausagem, é solicitada uma certeza de que o usuário realmente deseja realizar tal ação.

    • System Erros

      • Durante a mudança de estado, ou até mesmo durante o processo de start do sistema, há a possível chance de erros serem emitidos ao sistema, sendo eles:

      • User error - Erro por parte do usuário, com um comando incorreto, como usuário inválido ou comando escrito errado

      • Component failure - Falha direta com algum componente do hardware ou software do sistema.

      • Environment failure - Erro devido a recursos ou problemas de acesso, como buffer cheio ou algo do tipo.

      • Recoverable internal error - Erro de programação interna, não tão severo para abortar um programa.

      • Nonrecoverable internal error - Erro de programação interna, sendo severo o suficente para abortar um programa.

    • System Logs

      • Todos os principais componentes de software que rodam no host, são associados aos chamados registros de logs, ou arquivos de log, ou seja, é onde serão registradas as ações realizadas no servidor Netezza em questão, estando salvos todas as execuções. Os quais possuem características, como:

      • Cada log consiste em arquivos armazenados em um diretório específico. Para os gerenciadores, é dado um log por gerencia e para os servidores, sendo um log por sessão. Esses logs possuem uma ID e data/hora de emissão como identificador.

      • Para todos os componentes do Netezza, o sistema cria um novo log a meia-noite se o mesmo estiver em constante atividade.

      • A cada 30 dias, o sistema remove todos os logs naõ acessados que estejam datados neste mesmo período.

      A partir desses logs, é possível a realização de backups e restores do servidor (através dos comandos nzbakcup/nzrestore), onde os registrodos contidos nesses logs serão restaurados em um servidor X a partir de sua data de emissão. Além disso, essas realizações também irão gerar logs para registro, sendo armazenado o dia da execução, o tempo e outros dados da restauração ou backup.
